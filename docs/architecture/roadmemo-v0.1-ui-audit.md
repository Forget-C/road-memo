# RoadMemo UI 巡检记录 v0.1

## 1. 目标

本文件用于记录 V0.1 阶段 UI 巡检结果，明确：

- 已经完成的系统化收口
- 本轮已修复的“开发态”问题
- 后续仍可继续优化的剩余项

- 字段级错误反馈

## 3. 本轮已修复问题

### 3.1 统计页占位交互痕迹

问题：

- 统计页存在两个 `onClick = {}` 的“本月”按钮
- 这类无动作按钮会暴露 demo 感

处理：

- 主 header 的“本月”改成状态徽标
- hero 区的“本月”改成状态徽标

结论：

- 占位动作不再伪装成可点击操作
- 页面语义更真实，减少“壳子功能”感

### 3.2 统计页文案偏技术化

问题：

- 存在“统计口径”“当前说明”这类更像内部占位的表达

处理：

- “统计口径”改为“当前范围”
- “当前说明”改为“后续能力”
- 说明文案改成更偏用户理解的产品语言

结论：

- 统计页更像面向用户的报表页
- 减少技术化、占位化措辞

### 3.3 车辆页默认车识别不够强

问题：

- 默认车辆虽已存在，但概览和列表里的身份感还不够强

处理：

- 概览区补了“车辆数量 / 默认车辆”迷你信息卡
- 默认车辆在列表里统一改为状态徽标
- 默认车辆说明文案更明确地连接首页、记录、统计和提醒

结论：

- 车辆页的信息层级更清楚
- 默认车辆的重要性表达更稳定

### 3.4 首页与设置页说明文案偏长

问题：

- 首页和设置页部分说明文字仍偏“说明书口吻”
- 句子略长，产品语气还可以更轻一点

处理：

- 首页 header 和主卡说明压缩为更短的用户语言
- 设置页的概览、提醒、数据管理、车辆管理文案改成更直接的表达
- 单位与货币说明补齐当前版本边界

结论：

- 首页和设置页更接近日常工具产品的口吻
- 页面信息密度更稳定，不容易显得“开发文案”

### 3.5 主导航页 header 结构进一步统一

问题：

- 设置页仍沿用隐藏业务页 top bar 结构
- 记录页首屏被“当前车辆”卡和长按钮占用过多空间
- 首页、记录页、统计页 header 说明偏长

处理：

- 设置页切回主导航页 `ScreenHeader` 结构
- 记录页把“当前车辆”并入 header 描述，把主动作上提到 trailing 区
- 首页、记录页、统计页 description 压缩为更短的一行文案
- 首页 hero 区“切换默认车辆”缩短为“切车”
- 底部导航做了一轮轻量视觉收口，减少默认组件的刻板感

结论：

- 主导航页开场结构更一致
- 记录页首屏信息密度更高，可用空间更充足
- 首页和统计页 header 不再占用过多垂直空间

### 3.6 底部导航默认感过强

问题：

- 底部导航仍然偏默认 `NavigationBarItem` 风格
- 选中态比较刻板，和当前准扁平设计系统不够统一

处理：

- 底部导航改成更轻的自定义底栏结构
- 保持 4 个一级入口不变
- 选中态收口为图标胶囊 + 更清晰的文字层级
- 整体容器圆角和留白更柔和，减少默认组件感

结论：

- 底部导航更像正式产品底栏
- 视觉语言和当前的 card / pill / badge 系统一致性更高

### 3.7 设置页 header 顶部距离偏大

问题：

- 设置页虽然已经切回主导航页 header 结构
- 但外层 `Scaffold` 仍额外带入了顶部 inset
- 导致 header 整体比首页、记录、统计页更靠下

处理：

- 收掉设置页外层 `Scaffold` 的顶部 inset
- 统一首屏顶部内容距离为主导航页同一节奏

结论：

- 设置页 header 与其他一级页的顶部位置已对齐
- 主导航四页的开场结构进一步统一

### 3.8 设置页概览卡价值偏低，卡内布局偏竖向

问题：

- 设置页顶部“数据与提醒概览”更像说明型占位卡，缺少实际操作价值
- 一部分 tag 和次级按钮采用竖向堆叠，造成空间利用率偏低

处理：

- 去掉设置页顶部低价值概览卡
- 保留迷你状态卡作为首屏有效信息
- 将“显示与主题”“数据管理”里的 tag 改成横向排布
- 将“导出 CSV / 导出备份”收口为横向双按钮
- 将“提醒 / 车辆 / 恢复备份”统一为尾部横向动作

结论：

- 设置页首屏更聚焦有效信息
- 卡片内部信息密度更高
- 横向排布更符合当前工具型产品的准扁平规则

### 3.9 提醒卡片竖向堆叠偏多

问题：

- 提醒卡片里的 badge、来源动作和处理动作存在上下分层过多的问题
- 卡片高度偏高，列表扫描效率一般

处理：

- 将 badge 与来源动作收口到同一行
- 将“查看来源”缩短为“来源”
- 保留“忽略 / 已处理”作为右侧尾部动作组

结论：

- 提醒列表更紧凑
- 卡片更像正式待办处理卡，而不是说明型卡片

### 3.10 统计页 hero 区胶囊文本易换行

问题：

- 统计页 hero 区的状态胶囊中，`默认车辆` 文案在窄屏下容易挤压换行
- 胶囊组也缺少横向兜底策略

处理：

- 将 `默认车辆` 缩短为 `默认`
- 将 hero 区胶囊组改为横向滚动容器

结论：

- 统计页 hero 区更稳定
- 小屏下不再因为单个胶囊文案过长而破坏节奏

## 4. 当前仍可继续优化的残项

### 4.1 文案语义

仍有一部分技术化文案可继续产品化，例如：

- 统计页的分类行标题仍偏数据导向

可在后续统一改得更像面向用户的表达。

### 4.2 车辆页信息密度

车辆页已经补强默认车辆识别，但还可以继续增强：

- 多车状态的差异化信息
- 车辆卡的品牌感表达

### 4.3 统计页图表表达

当前趋势和分类仍是轻量表达，不是最终图表。

后续可继续：

- 接真实图表组件
- 做更完整的趋势线 / 占比图

### 4.4 首页摘要卡个性化

首页已经具备正式结构，但未来可进一步增强：

- 车辆昵称
- 更自然的月度摘要文案
- 更强的连续记账反馈

---

## 5. 当前结论

RoadMemo 当前 UI 已经从“功能原型”进入“可交付 MVP”的视觉阶段。

后续优化应以：

- 收尾一致性
- 提升产品感
- 避免回退到 demo 风格

为主，而不是继续大范围重做。

同时当前项目阶段也已切换为：

- 主线功能开发已完成
- UI 进入巡检收尾与发版前 polish 阶段

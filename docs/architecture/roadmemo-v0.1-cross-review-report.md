# RoadMemo V0.1 文档交叉审校报告

本报告用于交叉审校以下内容：

- 原始 PRD 与 Android 架构设想
- `roadmemo-v0.1-data-model.md`
- `RoadMemoDomainModelDraft.kt`
- `roadmemo-v0.1-freeze-checklist.md`

目标：

- 找出三份本地文档之间是否一致
- 标记与最初 PRD 的差异
- 提炼仍需确认的阻塞项

## 2. 总体结论

当前文档整体已经进入 **可冻结候选** 状态。

交叉审校结论如下：

- **核心建模已基本对齐**
- **与原始 PRD 存在少量有意收敛和重构**
- **仍有 3 个建议在建项前补充确认的点**

如果只看“能不能开始搭建项目”，答案是：

如果看“是否还建议最后做一轮文档收口”，答案是：

## 3. 已对齐的部分

### 3.1 产品定位与数据模型已对齐

当前文档已经稳定支持以下核心闭环：

- 多车管理
- 能源记录
- 保养记录
- 其他费用
- 续期事项
- 提醒
- 首页摘要
- 统计与导出备份

这和原始 PRD 的核心方向是一致的，只是对“加油记录”做了更合理的抽象升级。

### 3.2 多能源车型支持已对齐

原始 PRD 里车辆燃料类型包含：

- 汽油
- 柴油
- 混动
- 纯电

当前数据模型进一步细化为：

- `GASOLINE`
- `DIESEL`
- `HEV`
- `PHEV`
- `EV`

这是合理增强，不是偏离。

### 3.3 统计目标与聚合模型已对齐

原始 PRD 关注：

- 本月总支出
- 分类支出
- 油耗趋势
- 单车成本对比
- 每公里成本

当前文档已经有：

- `MonthlyCostSummary`
- `CategoryCostSummary`
- `EnergyConsumptionPoint`
- `VehicleComparisonSummary`
- `HomeSummary`

在领域建模上已经能支撑这些页面目标。

### 3.4 提醒能力已比原始 PRD 更完整

原始 PRD 只描述了提醒类型和提醒维度。  
当前文档进一步补了：

- 提醒状态流转
- 来源记录关联
- Android 通知权限约束
- 非精确提醒预期

这是明显增强，方向正确。

### 3.5 备份导出能力已补足

原始 PRD 提到：

- CSV 导出
- 本地备份恢复

当前文档已经补出：

- CSV 范围
- JSON 备份格式
- 覆盖式恢复策略

这部分已经比最初的描述更可实施。

## 4. 与原始 PRD 的有意差异

这些不是问题，而是 **为了支持纯电 / 插混和降低返工风险而做的结构升级**。

### 4.1 “加油记录”升级成“能源记录”

原始 PRD 里核心是“加油记录”。  
现在改成 `EnergyRecord`，统一承载：

- 加油
- 充电

这是本次文档升级中最正确的一步。

### 4.2 保险 / 年检从普通费用中拆出

原始 PRD 把保险、年检放在“其他费用记录”里。  
当前文档改为：

- 普通费用走 `expense_records`
- 有有效期的事项走 `renewal_records`

这是有意偏离，但更合理。

### 4.3 车辆表去掉 `currentMileage`

原始架构草案中 `VehicleEntity` 有 `currentMileage`。  
当前文档不再持久化该字段，而是改成派生值。

这是合理收敛，可以减少里程冲突。

### 4.4 单价不作为数据库权威字段

原始 PRD 中“单价 / 总金额二选一或反推”偏向表单逻辑。  
当前数据模型明确：

- 存总金额
- 存数量
- 单价由展示层反推

这也是合理收敛。

### 4.5 V1 目标用户范围收窄

原始 PRD 包含：

- 网约车司机
- 商务司机
- 二手车买家

冻结清单建议收敛为：

- 普通私家车主
- 家庭多车用户

这属于刻意降 scope，适合单人开发。

## 5. 当前文档之间仍存在的轻微不一致

### 5.1 首页摘要还没显式体现续期事项

当前：

- `MonthlyCostSummary` 已包含 `renewalCost`
- 但 `HomeSummary` 没有单独的“最近续期事项”或“即将续保事项摘要”

这不是阻塞项，但如果首页要展示：

- 最近保险
- 最近年检
- 即将续保提示

后续可能会再补一个聚合字段。

建议：

- 不改核心模型
- 在 UseCase 层通过 `upcomingReminders` 聚合展示即可

### 5.2 分类统计的 key 仍然偏松

`CategoryCostSummary` 当前是：

- `categoryType`
- `categoryKey: String`

这已经比纯字符串更好，但仍然允许错误 key 混入。

建议：

- 先不阻塞开工
- 后续在实现统计查询时统一由 mapper 输出规范 key

## 6. 已补齐并关闭的事项

以下事项已在最新文档中补齐：

1. 删除车辆时采用级联硬删除全部关联数据
2. `renewal_records` 自动创建和维护来源提醒
3. 记录页一级入口冻结为：能源 / 保养 / 费用 / 续期

因此，上一轮审校报告中的 3 个阻塞前事项已关闭。

## 7. 建议的最终冻结结论

如果没有新的业务分歧，建议直接采用以下最终冻结结论：

1. 续期事项统一进入 `renewal_records`
2. 删除车辆时采用级联硬删除全部关联数据，并弹确认提示
3. 续期事项自动维护来源提醒
4. 记录页主入口统一为：能源 / 保养 / 费用 / 续期
5. V1 不支持复杂历史补录重排
6. V1 不支持 odometer-only 记录
7. V1 提醒采用 `WorkManager` 近似触发
8. V1 备份恢复采用覆盖式恢复
9. V1 用户收敛为普通私家车主与家庭多车用户

## 8. 审校结论

当前文档质量已经足够高，结构性风险基本已消除。  
剩余工作更偏向实现阶段的细化，而不再是产品建模层面的返工风险。

一句话总结：

> RoadMemo V0.1 已经具备进入项目搭建前冻结的条件，可以作为冻结候选版本进入建项准备阶段。

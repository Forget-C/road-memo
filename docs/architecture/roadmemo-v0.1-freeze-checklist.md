# RoadMemo V0.1 文档冻结清单

## 1. 目的

这份清单用于在正式搭建项目之前，确认 RoadMemo V0.1 的产品边界、数据口径、平台约束与实现范围。

满足以下条件后，文档即可视为冻结：

- 核心数据模型不再调整
- 首版功能边界不再扩张
- 统计口径与提醒口径已定稿
- Android 平台限制已被产品接受

## 2. 推荐冻结结论

建议将 V0.1 冻结为以下版本：

- 面向普通私家车主
- 支持燃油车、纯电车、插混车、油混车
- 支持多车管理
- 以本地存储、本地提醒、本地备份为主
- 不做账号、不做云同步、不做 OCR、不做 GPS 自动里程

一句话定位：

> 一个支持多能源车型的本地优先车辆成本记录与提醒 App。

## 3. 必须拍板的 6 个决策

### 3.1 续期事项建模

决策项：

- 保险
- 年检
- 车船税

是否全部进入 `renewal_records`

推荐：

- **是**

原因：

- 这三类数据天然有“到期日”
- 能直接驱动提醒
- 避免落在普通费用里后再重复录提醒

冻结结论：

- `renewal_records` 承载续期类事项
- `expense_records` 不再承载保险、年检、车船税

### 3.2 历史补录重排

决策项：

- V1 是否支持用户补录很早以前的记录，并自动重排里程链路与能耗区间

推荐：

- **否**

原因：

- 会显著增加校验、重算、冲突处理复杂度
- 对 MVP 留存帮助不大

冻结结论：

- V1 不支持复杂历史补录重排
- 编辑旧记录若影响后续连续性，则阻止保存并提示用户

### 3.3 odometer-only 记录

决策项：

- V1 是否支持“只记录里程，不记录费用/能源/保养”的独立里程记录

推荐：

- **否**

原因：

- 会引入新的记录类型和里程口径
- 会增加首页派生里程、统计与提醒逻辑的复杂度

冻结结论：

- V1 不支持 odometer-only 记录
- 里程只附着在业务记录上

### 3.4 提醒触发精度

决策项：

- V1 是否承诺精确到分钟的提醒触发

推荐：

- **否**

原因：

- Android 13+ 通知权限需要显式授权
- Android 14 对 exact alarm 更严格
- 本项目场景更适合 `WorkManager` 近似提醒

冻结结论：

- V1 采用近似提醒
- 不承诺分钟级精确触发
- 产品文案避免使用“准时”“精确到点”等表述

### 3.5 备份恢复策略

决策项：

- V1 恢复备份时，是否支持与现有数据 merge

推荐：

- **否**

原因：

- merge 需要冲突检测、去重策略、版本兼容
- MVP 阶段最稳的是全量恢复

冻结结论：

- V1 采用“覆盖式恢复”
- 恢复前明确提示将覆盖当前本地数据

### 3.6 目标用户范围

决策项：

- V1 是否覆盖网约车司机、商务司机、收入核算、路线追踪场景

推荐：

- **否**

原因：

- 这些场景通常需要 `trip / route / income / business purpose`
- 当前数据模型未设计这些对象

冻结结论：

- V1 主用户为普通私家车主与家庭多车用户
- 网约车 / 商务用车场景放入后续版本

### 3.7 删除车辆策略

决策项：

- 删除车辆时，是否允许保留其历史记录

推荐：

- **否**

原因：

- 历史记录全部依附于车辆
- 保留孤立记录会增加筛选、统计和数据一致性复杂度

冻结结论：

- V1 删除车辆时采用级联硬删除
- 删除前弹出明确确认提示

### 3.8 续期事项与提醒联动

决策项：

- `renewal_records` 是否自动生成和维护来源提醒

推荐：

- **是**

原因：

- 续期事项天然就是提醒来源
- 手动双录入会显著降低体验并增加数据分叉概率

冻结结论：

- `reminder_enabled = true` 时自动创建来源提醒
- 修改到期日时同步更新提醒
- 关闭提醒时禁用来源提醒，不直接删除
- 删除续期事项时同步删除来源提醒

### 3.9 记录页入口结构

决策项：

- 记录页一级入口是否升级为“能源 / 保养 / 费用 / 续期”

推荐：

- **是**

原因：

- 当前模型已将能源与续期独立建模
- 延续旧的“加油 / 保养 / 其他费用”结构会和数据层不一致

冻结结论：

- 记录页一级入口固定为：
  1. 能源
  2. 保养
  3. 费用
  4. 续期

## 4. 已冻结的核心口径

### 4.1 数据对象

V0.1 保留以下核心对象：

- `Vehicle`
- `EnergyRecord`
- `MaintenanceRecord`
- `ExpenseRecord`
- `RenewalRecord`
- `Reminder`

### 4.2 车型与能源模型

- 车辆动力类型：`GASOLINE / DIESEL / HEV / PHEV / EV`
- 能源类型：`FUEL / ELECTRIC`
- 插混车首版只做油、电分开统计
- 不做综合等效能耗

### 4.3 金额与数量精度

- 金额统一用分：`amount_in_cent`
- 能源数量统一用千分位整数：`quantity_in_thousandth`
- 数据库存储不使用 `Double` 保存金额

### 4.4 里程口径

- `currentMileage` 不入表，作为派生值
- 同一车辆录入时里程不能回退
- V1 不支持复杂历史补录重排

### 4.5 能耗口径

- 仅相邻两条 `is_full = true` 的同能源记录形成有效区间
- `is_full = false` 仅参与费用统计，不参与区间终点计算
- 编辑或删除历史能源记录后，后续区间需重算

### 4.6 续期与提醒口径

- 保险、年检、车船税进入 `renewal_records`
- 提醒有生命周期状态，不是一次性布尔开关
- 来源记录删除时，关联提醒同步删除

## 5. Android 平台接受项

产品需要接受以下平台现实：

- Android 13+ 新安装应用默认没有通知权限
- 用户拒绝通知权限后，提醒记录仍存在，但通知无法真正送达
- V1 采用 `WorkManager` 进行提醒扫描与下发
- V1 不依赖 exact alarm
